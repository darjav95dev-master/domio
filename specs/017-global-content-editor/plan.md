# Implementation Plan: global-content-editor

**Branch**: `feature/017-global-content-editor` | **Date**: 2026-07-08 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification from `/specs/017-global-content-editor/spec.md`

## Summary

Implementar un editor de contenidos comerciales globales para el backoffice de Domio que permita a ADMIN y OPERATOR editar bloques de contenido por página (home, sobre, equipo, legales), gestionar la configuración de contacto global, y navegar/revertir el historial versionado de cambios. El sistema valida payloads JSONB con schemas Zod específicos por tipo de bloque, persiste en tablas existentes (content_blocks, contact_config, content_history), dispara revalidación ISR tras cada cambio, y respeta aislamiento multi-tenant con RLS.

## Technical Context

**Language/Version**: TypeScript 5.x (strict mode)

**Primary Dependencies**: Next.js 15 (App Router), Drizzle ORM, Zod, Tailwind CSS v4, Auth.js v5

**Storage**: PostgreSQL 16 (Neon) con RLS activado. Transaction pooling con `SET LOCAL app.tenant_id` obligatorio dentro de transacciones.

**Testing**: Vitest (unit/integration), Playwright (E2E), @testing-library/react (componentes)

**Target Platform**: Web (Vercel hosting) — SSR/ISR en superficies públicas, Server Components en backoffice

**Project Type**: Web application (SaaS multi-tenant inmobiliario)

**Performance Goals**: Revalidación ISR < 60s tras guardar cambios. Edición de bloques < 30s por bloque (tiempo de interacción humana).

**Constraints**: Payloads JSONB validados con Zod antes de persistir. content_history es inmutable (RLS impide UPDATE/DELETE). Revalidación automática de páginas públicas afectadas.

**Scale/Scope**: 6 páginas editables (home, sobre, equipo, aviso-legal, privacidad, cookies), múltiples bloques por página, 1 configuración de contacto global por tenant, historial versionado infinito.
